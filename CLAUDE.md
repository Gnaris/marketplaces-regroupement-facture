# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Chrome MV3 extension (vanilla JS, no build step, no tests, no package manager). On an order-details page of one of six B2B fashion marketplaces, it adds a "Regrouper la facture" button that fetches the order via the site's API and renders an aggregated table grouped by `(catégorie, prix unitaire)` plus the shipping / billing addresses extracted from the order.

Supported sites (`site` codes used throughout the code):
- `pfs` — wholesaler.parisfashionshops.com
- `efp` — wholesaler.efashion-paris.com
- `aks` — *.ankorstore.com
- `faire` — www.faire.com (brand-portal / seller side)
- `mc` — web.mc.app + mc2-h5.dokkr.net (Microstore — seller-side B2B order portal)
- `oc` — www.orderchamp.com (Orderchamp — supplier backoffice)

## Developing & loading

There is no build. To test changes: `chrome://extensions` → enable Developer mode → "Load unpacked" → select this folder. After editing, click the reload icon on the extension card; for `content.js` / `modal.css` changes also reload the order page. For `background.js` changes use the "service worker" link on the card to inspect the worker.

Bump `manifest.json` `version` when shipping.

## Architecture

Three layers, talking via `chrome.runtime` messages and (for PFS only) `window.postMessage`:

1. **`content.js`** (runs on each order page at `document_start`) — detects `site` from `location.hostname`, extracts `orderId` from the URL with a site-specific regex, injects the floating button, and on click sends a `FETCH_ORDER` message to the background. A `MutationObserver` re-injects the button after SPA navigations. Parsing of the API response and modal rendering both live here.

2. **`background.js`** (service worker) — single `chrome.runtime.onMessage` listener that branches on `message.site` and calls the right endpoint:
   - `pfs` → `GET https://wholesaler-api.parisfashionshops.com/api/v1/orders/{id}` with the captured `Authorization` header.
   - `efp` → `POST https://wapi.efashion-paris.com/graphql` with the inlined `EFP_QUERY` (`commandeById`), `credentials: "include"`. The query also pulls `acheteur` (contact name, VAT, EORI, email) and `adresseFacturation` (street, postal code, city, country) so `parseEfp` can build the billing block.
   - `aks` → **two parallel fetches** via `Promise.all`: `/order-items?include=orderedProduct` for items and `/orders/{id}` (no path suffix) for the shipping address and retailer business info (VAT / tax number). The response is `{ items, detail }`; `parseAks` consumes both. The `apiBase` is `location.origin` from the content script, so it follows the user's country subdomain. `credentials: "include"`.
   - `faire` → `GET https://www.faire.com/api/brand-order/{id}/details`, `credentials: "include"`. The `id` is a brand-order token like `bo_xxx`. **Addresses are not in this response** — they come from a separate `customer_details` call intercepted by `injected-faire.js` (see below).
   - `mc` → **does not fetch anything**. Microstore's order-detail endpoint (`api2.dokkr.net/index.php/pluginsWeb/orderInfo/{id}?key=...`) requires a per-session `key` query param that's only generated by the host app at runtime, and the order detail opens as a side panel without changing the URL. Instead we **intercept** the response in the page world via `injected-mc.js` (monkey-patches `fetch` + `XHR`), `postMessage` it back to `content.js`, and the content script forwards it to `background.js` via `SAVE_MC_ORDER`. The service worker caches the latest response per `sender.tab.id` in `lastMcOrders` (in-memory `Map`, cleared on tab close). When the user clicks the floating button, the content script asks the worker for the cached payload via `GET_MC_ORDER`. So the worker never reaches the Microstore API itself.
   - `oc` → **does not fetch anything either**. Orderchamp's order detail page is server-rendered HTML, so `parseOc` reads the products and addresses directly from the DOM at click time. No background round-trip needed.

   The host's API origins are declared in `manifest.json` `host_permissions` so the service worker can hit them; without that, the fetch is blocked.

3. **`injected.js`** (PFS only, declared in `web_accessible_resources`) — runs in the **page world** (content scripts run in an isolated world and cannot read the page's in-memory auth token). It monkey-patches `window.fetch` and `XMLHttpRequest` to sniff the `Authorization` header on calls to `wholesaler-api.parisfashionshops.com`, then forwards it to the content script via `window.postMessage({ source: "pfs-regroup", type: "AUTH_TOKEN", token })`. The content script caches this token and forwards it to the background. EFP and Ankorstore don't need this because their APIs accept cookie auth via `credentials: "include"`.

   Consequence: on PFS, the user must have triggered at least one API call (typically by loading the page while logged in) before the button works. The error message "Connexion non détectée. Rechargez la page (F5)…" is the surface for this.

4. **`injected-mc.js`** (Microstore, declared in `web_accessible_resources`) — same page-world injection pattern as `injected.js`, but it sniffs the **full response body** of `pluginsWeb/orderInfo/` calls rather than a header. The Microstore content script is declared with `all_frames: true` in `manifest.json` because the API request happens inside an iframe (`mc2-h5.dokkr.net`) embedded in `web.mc.app` — both frames need the sniffer, but only the top frame (`web.mc.app`) shows the button (`if (!isTopFrame) return;` after wiring up the listener). Site detection accepts both hosts: `host === HOST_MC || host.endsWith(".dokkr.net")`.

5. **`injected-faire.js`** (Faire, declared in `web_accessible_resources`) — same pattern again, but for Faire's `/api/v3/crm/{brandToken}/customer_details?retailer_token={retailerToken}` endpoint. Faire's brand-order endpoint does NOT include the retailer's addresses, but the brand portal page itself triggers `customer_details` when an order is opened — we intercept that response and cache it in `cachedFaireCustomer` (a content-script-local variable, not in background). `parseFaire(json, customerData)` uses the cached customer to build the shipping (`default_shipping_address`) and billing (`retailer_details.address`) blocks. Like PFS, this means **the order page must be loaded once before the button works**; otherwise the addresses are silently omitted and a `console.log` warns to reload.

## Aggregation contract

All six parsers normalize into the same shape consumed by `showModal`:

```js
{ rows: [{ categorie, prixUnitaire, quantite }, ...],
  orderInfo: {
    orderNo,
    totalVerif,
    addresses: [
      { title: "Adresse de livraison" | "Adresse de facturation",
        fields: [{ label, value }, ...] }
    ]
  } }
```

Rows are merged via `addRow(map, cat, price, qty)` using the key `cat + "||" + price.toFixed(4)` — so two lines with the same category but a 0.0001 € price difference become two rows. If you change rounding here, also change it everywhere prices are read.

Addresses are built field-by-field via `pushField(fields, label, value)` (skips empty values, trims whitespace). The modal renders each block as a `<dl>` with `<dt>label :</dt><dd>value</dd>` rows in a 140px-label grid. A field with `label: null` is rendered as a single italic note across the row — currently used for MC's billing placeholder (`Voir l'iPad`, since Microstore's order endpoint does not expose billing). Per-site address policy:
- `pfs`, `efp`: **billing only** (shipping is on the shipping label PDF the platform already produces, so it would be redundant).
- `aks`: **shipping only** (built from `data.attributes.shipping.shippingAddress` + `retailer.business` for VAT / tax number).
- `mc`: **shipping** from `client_info`, **billing** = literal note "Voir l'iPad" (the user explicitly requested this — the endpoint doesn't reliably expose billing).
- `faire`: **shipping** (`default_shipping_address`) + **billing** (`retailer_details.address`), both from the intercepted `customer_details` response.
- `oc`: **shipping + billing**, parsed from the DOM by `parseOcAddresses` (walks siblings after `<strong>Adresse de livraison/facturation</strong>` until `<hr>` or next `<strong>`, then `ocLinesToFields` heuristically classifies lines: special `Numéro de TVA:` / `Numéro d'enregistrement:` are extracted first, then a `^(\d{2,6}[A-Za-z]{0,3})\s+(.+)$` regex finds the postal-code+city line, lines before it become company / contact / address, line after becomes country).

Per-site quirks the parsers handle:
- **PFS** `parsePfs`: walks `items_by_brand[].products[].items[]`. Items with `type === "PACK"` count as `qty_validated * pieces`. Category is `product.category.labels.fr`.
- **EFP** `parseEfp`: skips `ligne.corbeille === 1` (trash). Price prefers `prixReduit` when > 0, falls back to `prix`. Category comes directly on the ligne.
- **AKS** `parseAks`: Ankorstore has no category field, so `detectCategory(name)` infers one from the product name with French regexes (boucles d'oreilles, créoles, bracelet jonc/charm, bague d'orteil/ajustable, etc.). Prices arrive as cents in `unitPrice.amount` — divide by 100. Products are in `included[]` with `type === "ordered-products"`, joined to items via `relationships.orderedProduct.data.id`. When adding a new heuristic, put more-specific patterns before more-general ones (the function returns on first match).
- **Faire** `parseFaire`: walks `items[]`. Skips items where `is_sample`, `includes_tester`, `state === "CANCELED"`, or `order_item_type !== "PRODUCT"`. Price = `wholesale_price.amount_cents / 100` — this is the B2B wholesale price the retailer pays (Faire is wholesale-side; `retail_price` is the suggested resale RRP, not what's charged). Quantity = `item_quantity * (unit_multiplier || 1)`. **Category is derived via `detectCategory(item.product_name)`, NOT from Faire's native `products[token].taxonomy_type.name`** — the user explicitly rejects the native taxonomy because Faire's labels ("Solitaire", "Anneau/bague à empiler", "Bague de cocktail/de cérémonie"…) don't match his internal classification. Order number for display: `brand_order.token` with the `bo_` prefix stripped and uppercased. Verification total: `item_subtotal.amount_cents / 100`. The `samples/faire-ordre.json` file is a real captured response used for offline testing; it's not used at runtime.
- **Microstore** `parseMc`: walks `json.data.doc_info.goods_info[]`. Skips items where `disable === "1"` or `status === "0"`. Numeric fields all arrive as **strings** so `parseFloat` / `parseInt` are required: `price = parseFloat(item.price)`, `qty = parseInt(item.quantity, 10)`. **Category is derived via `detectCategory(item.name)`** (same rationale as Faire — Microstore's `category_name` like "Lot de bracelets avec présentoir" is intentionally ignored). Order number = `doc.number`. Verification total = `parseFloat(doc.total_price)`. The JSON comes from the intercepted `pluginsWeb/orderInfo/{id}` response (not a direct fetch — see `injected-mc.js` above).
- **Orderchamp** `parseOc`: runs on the order detail page (`/{locale}/backoffice/orders/{numericId}`) and reads the DOM directly — no fetch. Finds every `<a href*="/backoffice/products/">`, walks up to the enclosing `<tr>`, dedupes by row (the row has both an image `<a>` and a text `<a>` pointing to the same product), then locates the cell whose text matches `\d+\s*x\s*[\d\s.,]+\s*€` to extract the quantity and the **discounted** unit price (`text-striked-through` siblings are present but the regex captures the first euro amount only). **Category is derived via `detectCategory(name)`** (consistent with Faire/MC; Orderchamp doesn't expose a usable category). Order number is read from the page `<title>` (`/OC\d+/`). Verification total is the `<tr>` whose first `<td>` text equals `"Total"` in the totals table. Helper `parseEuroAmount` normalizes both `"5,40 €"` (decimal comma) and `"1 234,56 €"` (thousands space).

Note on `detectCategory` regexes: they must handle both apostrophes (`'` U+0027 and `’` U+2019 — Faire's `product_name` uses the typographic variant) and plurals (e.g., `cha[îi]nes?` matches both "Chaîne" and "Chaînes"). When adding new marketplaces, prefer `detectCategory(name)` over any native category field for consistency with the user's classification.

## UI

`modal.css` is loaded via `manifest.json` `content_scripts.css` and uses `#pfs-*` / `.pfs-*` IDs and classes throughout to avoid colliding with the host page. The button and overlay use `z-index: 2147483646/7` for the same reason. DOM is built imperatively via the local `el(tag, props, children)` helper in `content.js` — no framework.
